1 what it is

  1. what it is
    • a voice concierge on the hotel's website
    • the guest taps the mic and just talks — no typing
    • German or English; it answers like a receptionist
  2. what it does
    • shows matching rooms as photo cards
    • checks live availability + prices
    • hands off a secure payment link
    • puts staff phone + email on screen when it can't help
  3. the one rule
    • every room, price and date comes from a live lookup that same turn
    • it never invents one — a wrong price is a real liability

2 how it works

guest  ◀─voice─▶  gpt-realtime call  ◀─tools─▶  our MCP tools ×9  ◀─live─▶  booking · site · staff
       speaks         hears + speaks                data, code renders          (sources of truth)
  1. the guest talks to the page
    • opening the panel opens a live voice call straight to OpenAI (gpt-realtime, voice marin)
    • one model hears, thinks and speaks — German or English, interruptible mid-sentence
    • if the call can't connect it drops silently to a tap-to-talk fallback — same tools, same cards
  2. the model can't answer from memory — it calls a tool
    • rooms, prices and facts are never in the model's head; it has to ask a tool
    • our server runs the tool and returns both the answer and what to show on screen
    • the 9 tools it can call — and how each is kept honest — are in the MCP tools tab →
  3. the screen shows, the voice tells
    • cards, prices, links and contact buttons are drawn by code from the tool's result
    • so the panel literally cannot display a made-up price
three rules that never bend:
  1. only from a tool — a room / price / date / fact is said only if a tool returned it that turn
  2. prices match exactly — the on-screen cards carry the tool's exact number on both paths, and on the HTTP fallback the spoken text is code-locked to it too; spoken call-mode prices are prompt-guarded, with the code-level check still open (§10)
  3. new abilities = new tools — the product grows without rewriting the brain

2b the prompts that steer her

two brains, two prompts
  • call mode: the persona in server/realtime-instructions.js — 9 labelled sections (Role · Personality & Tone · Tools · Values · Rules · Unclear Audio · Flow · Language · Escalation)
  • fallback mode: the system prompt in server/prompt.js — same Maya identity, plus a {{Pn}} price-marker rule so code swaps in the exact numbers
what she starts a call with
  • a fixed greeting, spoken once, live: "Hi, welcome to Park Hotel Konz — I'm Maya…"
  • on a reconnect it swaps to "Welcome back — where were we?" — she never greets twice
the rules baked into both prompts
  • never invent a price / date / fact — only say what a tool returned this turn
  • every reply is 100% in the guest's language
  • never claim to be human — she's Maya, the hotel's digital assistant
the other prompts in play
  • a booking-requirements block (auto-generated) — the "always ask dates + party size" rule, shared by the prompt and the tool schemas
  • a local-context line (Konz time + weather) injected at the start of each call to steer small talk
  • the attractions-curator prompt that runs the 48 h freshness loop — see the MCP tools tab →

3 the voice

  1. what she runs on
    • a real speech-to-speech model — OpenAI gpt-realtime, voice marin
    • one model hears, thinks and speaks, so warmth, laughter and DE/EN switching are natural, not bolted on
  2. why this and not an ordinary text-to-speech voice
    • a chained TTS voice performs a flat script; a speech-to-speech model carries its own tone across the whole call
    • the client's bar: a disclosed AI that still feels human
  3. she's "Maya", and always honest about being digital
    • she greets as Maya; asked outright, she says she's the hotel's digital assistant — never claims to be a person
  4. the screen beats reciting
    • she never spells an email or reads a long number aloud — she points to the on-screen button or card
    • prices are spoken only when the guest explicitly asks; otherwise she describes the room and points to the rate
  5. if the call fails, nothing breaks
    • a tap-to-talk fallback (Voxtral → Mistral → OpenAI TTS) takes over silently — same tools, same cards
  6. privacy
    • call transcripts are used for live captions only and are never stored on our server
    • formal EU data-residency sign-off is an open owner decision (tab ① · decisions still needed)

the deep engineering detail — the delivery levers, the reverted pacing + reasoning-effort experiments, the language-hardening rules — lives in concierge-design.md §3, not on this page.

4 the booking engine (live prices)

  1. where live prices come from
    • the SiteMinder "Canvas" booking engine (direct-book.com/…/parkhotelkonzdirect), wired via mcp/lib/booking.js
    • called directly over plain fetch since 2026-07-08 — $0 per capture, and fast (seconds, not the old browser route's tens of seconds)
    • if the firewall ever blocks again, a paid Apify browser capture is the automatic fallback (last resort only)
  2. fast + cheap
    • 4 room quotes in flight at once; a 30-min cache answers instantly (serves stale up to 24 h) while it refreshes
    • opening the widget warms the next 7 days, so near-future asks answer with zero wait
  3. honest when it can't reach live
    • if both paths fail it says so plainly, still shows room cards + a Book link with dates pre-filled — never an invented price
    • every paid fallback emails the owner (throttled); /api/health shows which path served last
  4. payment
    • the guest pays on SiteMinder's own page via a pre-filled deep-link — card data never touches the chat
    • SiteMinder exposes no API, so the deep-link IS the payment mechanism for now

the tuning detail — cache TTLs, warm-sweep env dials, retry/backoff counts, the GraphQL IDs — lives in concierge-design.md §4 + hotel_information/booking-engine-api.md.

5 the MCP tools

  1. what MCP is
    • MCP (Model Context Protocol) is a standard way to give the AI hands — each ability is a typed "tool" it can call
    • the tool returns data; our code renders it; the model may only narrate what came back
  2. why we use it
    • grounded — every fact / price must arrive through a tool, so nothing is invented
    • auditable — every call + result is recorded (see the Testing examples tab)
    • reusable — the same 9 tools serve the live call, the fallback loop, and any MCP client
  3. the party-size guardrail
    • a booking tool refuses to run unless the guest actually stated dates + party size
    • enforced at the MCP boundary (mcp/registry.js) on the live-call + fallback transports; a raw stdio MCP client sends no transcript so it skips the invented-party check — but the required dates + party fields are enforced everywhere

the 9 tools — click any one to verify what it returns, when it's called, and how it's grounded.

search_rooms — live availability + prices for a stay
  • returns: a live booking-engine capture per room — available?, units left, total + per-night rate, photos → room cards
  • called when: open "what's free?" asks — only after the guest gave dates + party size
  • example: "what's available Jul 24–26 for 2?" → Comfort Double Plus €240 ✓, the rest sold out
  • grounded: nothing static — every number is from that live call; refuses without dates + party
check_availability — is one named room open, at what rate
  • returns: the same live capture filtered to one room, + a note if the party is too big for it → that room's card
  • called when: the guest names a room ("is the Basic Apartment free?")
  • example: "Standard-Doppelzimmer 24.–26., 2 Erw.?" → sold out → offers the available alternative from the same capture
  • grounded: live capture only; refuses without dates + party
get_room_details — one room's features (scraped)
  • returns: the room sheet — photos, size (m²), view, amenities (+ a live total if dates are given)
  • called when: feature questions ("does the apartment have a kitchen?") — no dates needed
  • example: "what's special about the Basic Apartment?" → 30 m², kitchen, balcony, 6 photos → card
  • grounded: scraped facts only; she describes only what the tool returned

  • example: "when is breakfast at the weekend?" → "8:00–10:00" — never from memory
  • grounded: the prompt holds NO facts — this tool is the only source, so every fact answer has provenance

how this list stays current — the 48 h loop
  • where the prompt lives: tools/attractions-curator.md, run by tools/run-curator.cmd — a Windows task (ParkHotel-AttractionsCurator) fires it every 48 h as a headless Claude Code run; log in logs/curator.log.
  • what it searches for: re-verifies entries not checked in 14+ days against their official site; hunts seasonal events in Konz · Trier · Saarburg · the Mosel valley for the next 3 months; spots permanent closures.
  • how it keeps data fresh: adds events with exact dates; removes passed events + confirmed closures; if it can't verify it does NOT guess (keeps the old value, marks "unverified since <date>"); validates JSON + tests, then opens a PR — never pushes master.
get_train_connections — live train departures from Konz
  • returns: live Deutsche Bahn departures from Konz station — line, scheduled time, live delay (min), platform, direction; filter by destination (Trier / Luxembourg / Saarburg)
  • called when: "how do I get to Trier by train?" / next departures
  • example: "trains to Trier?" → the next few departures with their live delays
  • grounded: every departure is live from this call; if the timetable is unreachable it returns degraded and she says so honestly — never an invented time
